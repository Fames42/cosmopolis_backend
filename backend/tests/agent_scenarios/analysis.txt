# Recommendations for improvement

## 1. Reduce scheduling friction
When the tenant says:
- “first available”
- “tomorrow morning”
- “after 18:00”
- “any time”
the agent should search immediately using broad preferences rather than forcing exact-time input first.

## 2. Standardize emergency workflow
For emergencies:
- give safety instructions
- use `update_service_details(...)`
- use `escalate_to_human(...)` explicitly
- create emergency ticket if required by policy
- avoid slot-search behavior unless the backend specifically uses a dedicated emergency dispatch path

## 3. Improve FAQ handling
If the system supports FAQ answers, the agent should answer directly when possible, and only escalate when:
- the information is unavailable,
- policy is uncertain,
- or the tenant requests a human.

## 4. Make phrasing more natural and less templated
Replace forms like:
- “Понял(а)”
- “Зафиксировал(а)”
with more natural alternatives:
- “Понял, спасибо”
- “Хорошо, записал”
- “Спасибо, уточнил”

## 5. Tighten category mapping for HVAC-type issues
Cases like “air conditioner not cooling” should ideally map to a clearer HVAC/cooling category if available. If only `heating` exists, the wording and internal taxonomy should still be reviewed to avoid semantic mismatch.

## 6. Clarify no-ticket cancellation messaging
When a tenant cancels before a ticket is created, the agent should explicitly say something like:
- “Хорошо, заявку не создаю.”
This makes the outcome clearer than simply closing the conversation.
